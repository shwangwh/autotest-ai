# 测试用例管理功能设计方案

## 1. 功能需求

为测试用例管理模块增加以下功能：
- **新增测试用例**：手动创建新的测试用例
- **编辑测试用例**：修改现有测试用例的信息
- **删除测试用例**：删除不需要的测试用例

## 2. 技术分析

### 2.1 现有代码结构

#### 后端
- `TestCaseService.java`：测试用例服务类，包含核心业务逻辑
- `TestCaseController.java`：测试用例控制器，处理HTTP请求
- `TestCase.java`：测试用例实体类，定义数据模型

#### 前端
- 测试用例列表页面：显示测试用例列表，包含"删除"操作
- 缺少新增和编辑测试用例的界面

### 2.2 现有功能分析

- **已实现的API接口**：
  - GET /api/test-cases - 获取所有测试用例
  - GET /api/test-cases/project/{projectId} - 获取项目的测试用例
  - GET /api/test-cases/test-point/{testPointId} - 获取测试点的测试用例
  - GET /api/test-cases/{id} - 获取单个测试用例
  - POST /api/test-cases/generate - 生成测试用例
  - POST /api/test-cases/generate/batch - 批量生成测试用例
  - PUT /api/test-cases/{id} - 更新测试用例
  - DELETE /api/test-cases/{id} - 删除测试用例

- **缺少的功能**：
  - 新增测试用例的API接口
  - 前端新增和编辑测试用例的界面
  - 前端删除测试用例的确认机制

## 3. 设计方案

### 3.1 后端设计

#### 3.1.1 新增测试用例功能

**1. 在TestCaseService中添加addTestCase方法**

```java
public TestCase addTestCase(TestCase testCase) {
    // 生成测试用例编号
    if (testCase.getCaseNumber() == null || testCase.getCaseNumber().isEmpty()) {
        String caseNumber = "TC-" + testCase.getTestPointId() + "-" + System.currentTimeMillis();
        testCase.setCaseNumber(caseNumber);
        testCase.setCaseCode(caseNumber);
    }
    
    // 设置默认值
    if (testCase.getStatus() == null) {
        testCase.setStatus("PENDING_REVIEW");
    }
    if (testCase.getPriority() == null) {
        testCase.setPriority("MEDIUM");
    }
    if (testCase.getAutomation() == null) {
        testCase.setAutomation(false);
    }
    if (testCase.getCreator() == null) {
        testCase.setCreator("admin");
    }
    
    return testCaseRepository.save(testCase);
}
```

**2. 在TestCaseController中添加新增测试用例的API接口**

```java
@PostMapping
public TestCase addTestCase(@RequestBody TestCase testCase) {
    return testCaseService.addTestCase(testCase);
}
```

### 3.2 前端设计

#### 3.2.1 测试用例列表页面优化

**1. 添加"新增测试用例"按钮**
- 在测试用例列表页面顶部添加按钮
- 点击按钮打开新增测试用例表单

**2. 优化"编辑"操作**
- 在测试用例列表的"操作"列添加"编辑"按钮
- 点击按钮打开编辑测试用例表单，预填充现有数据

**3. 优化"删除"操作**
- 点击"删除"按钮时，显示确认对话框
- 确认后执行删除操作

#### 3.2.2 测试用例表单设计

**1. 表单字段**
- 项目ID (projectId)：下拉选择
- 测试点ID (testPointId)：下拉选择
- 标题 (title)：文本输入
- 前置条件 (precondition)：文本域
- 测试步骤 (steps)：文本域
- 预期结果 (expectedResult)：文本域
- 优先级 (priority)：下拉选择 (HIGH, MEDIUM, LOW)
- 自动化 (automation)：复选框
- 状态 (status)：下拉选择 (PENDING_REVIEW, READY, EXECUTING, PASSED, FAILED)

**2. 表单验证**
- 标题不能为空
- 测试步骤不能为空
- 预期结果不能为空

**3. 提交处理**
- 新增：调用 POST /api/test-cases
- 编辑：调用 PUT /api/test-cases/{id}
- 成功后刷新列表，关闭表单

#### 3.2.3 确认对话框设计

**删除确认对话框**
- 显示删除确认消息
- 提供"确认"和"取消"按钮
- 确认后调用 DELETE /api/test-cases/{id}
- 成功后刷新列表

## 7. 风险评估

### 7.1 潜在风险
- 表单验证不充分，导致数据不完整
- 并发操作可能导致数据冲突
- 前端错误处理不完善，影响用户体验

### 7.2 应对措施
- 加强后端数据验证，确保数据完整性
- 实现乐观锁机制，处理并发操作
- 完善前端错误处理，提供友好的错误提示
