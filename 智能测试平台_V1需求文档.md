# 智能测试平台 V1 需求文档

## 1. 文档信息

- 文档名称：智能测试平台 V1 需求文档
- 文档版本：V1.0
- 文档状态：初稿
- 适用范围：产品、前端、后端、测试
- 当前阶段：第一版本需求定义

## 2. 项目背景

当前测试设计与执行过程存在以下问题：

1. 需求理解依赖人工阅读，测试点提取效率低。
2. 测试用例编写重复性高，规范性依赖个人经验。
3. 接口自动化执行与用例设计脱节，链路不闭环。
4. 测试结果与报告分散，缺少统一沉淀与追溯能力。

为提升需求分析、用例生成与接口自动化执行效率，计划建设一个基于 `Vue + Spring Boot + LLM` 的智能测试平台，第一版聚焦以下三大模块：

1. 需求设计
2. 用例设计
3. 接口自动化

Web 自动化放在第二期，不纳入 V1 范围。

## 3. 产品目标

V1 目标是打通从需求到接口自动化执行的核心闭环：

`上传需求文件 -> LLM 生成测试点 -> 选择测试点 -> LLM 生成测试用例 -> 选择测试用例 -> 接口自动执行 -> 返回结果 -> Allure 报告`

具体目标如下：

1. 支持上传需求文档并由 LLM 自动提取测试点。
2. 支持基于指定测试点生成测试用例。
3. 支持基于指定测试用例触发接口自动化执行。
4. 支持保存执行结果，并统一查看执行状态、结果详情和 Allure 报告。
5. 支持多项目管理，不同项目之间数据相互隔离。

## 4. V1 范围

### 4.1 包含范围

1. 登录页
2. 项目列表页
3. 项目内工作台
4. 需求设计模块
5. 用例设计模块
6. 接口自动化模块
7. 基础项目管理
8. 基础执行结果管理
9. Allure 报告查看入口

### 4.2 不包含范围

1. Web 自动化
2. 移动端自动化
3. 性能测试
4. 分布式执行
5. 智能自愈
6. 复杂流程编排
7. 细粒度组织架构权限

## 5. 用户角色

V1 可先按简单角色设计：

### 5.1 管理员

1. 登录系统
2. 查看全部项目
3. 创建和维护项目
4. 查看全部需求、用例、执行记录

### 5.2 测试人员

1. 进入被授权项目
2. 上传需求文件
3. 生成测试点
4. 生成测试用例
5. 执行接口自动化
6. 查看结果与报告

## 6. 核心业务流程

### 6.1 登录与进入项目

1. 用户打开平台，进入登录页。
2. 输入账号密码后登录系统。
3. 登录成功后进入项目列表页。
4. 用户点击某个项目，进入该项目工作台。
5. 进入项目后，左侧显示：
   - 需求设计
   - 用例设计
   - 接口自动化

### 6.2 需求到用例到执行的主链路

1. 用户在需求设计模块上传需求文件。
2. 平台调用 LLM 对需求内容进行分析，生成测试点。
3. 用户对测试点进行筛选、确认或编辑。
4. 用户在用例设计模块选择指定测试点。
5. 平台调用 LLM 基于测试点生成测试用例。
6. 用户确认或修订测试用例，并标记是否可自动化。
7. 用户在接口自动化模块选择指定测试用例。
8. 后端自动执行接口请求与断言。
9. 平台返回执行结果，并提供 Allure 报告查看入口。

## 7. 信息架构

## 7.1 页面层级

系统整体层级如下：

`登录页 -> 项目列表页 -> 项目工作台 -> 模块页面`

项目工作台内的左侧菜单为：

1. 需求设计
2. 用例设计
3. 接口自动化

### 7.2 交互规则

1. 用户未选择项目时，不展示模块列表页内容。
2. 用户必须先进入具体项目，才能看到三个模块。
3. 项目内页面左上角提供返回项目列表箭头。
4. 项目内数据均归属于当前项目。

## 8. 功能需求

## 8.1 登录

### 8.1.1 功能说明

提供系统基础登录入口。

### 8.1.2 功能要求

1. 提供账号、密码输入框。
2. 提供登录按钮。
3. 登录成功后进入项目列表页。
4. 登录失败时提示账号或密码错误。
5. V1 演示版本可先采用固定账号密码：
   - 账号：`admin`
   - 密码：`1`

### 8.1.3 后续扩展

1. 接入 Spring Security 或统一认证中心。
2. 支持真实用户体系和角色权限。

## 8.2 项目管理

### 8.2.1 功能说明

平台支持多个项目，所有需求、测试点、用例、接口执行记录均归属于项目。

### 8.2.2 功能要求

1. 登录后默认进入项目列表页。
2. 项目列表展示项目名称、项目简介、需求数量、用例数量、接口任务数量。
3. 支持点击项目进入项目工作台。
4. 支持返回项目列表。
5. 不同项目间的数据必须隔离。

### 8.2.3 列表字段建议

1. 项目名称
2. 项目描述
3. 需求数量
4. 用例数量
5. 接口任务数量
6. 创建时间
7. 更新时间

## 8.3 需求设计模块

### 8.3.1 功能目标

通过上传需求文件，调用 LLM 自动生成测试点，并沉淀为结构化列表。

### 8.3.2 页面形式

采用列表页结构，包含：

1. 顶部标题与面包屑
2. 条件筛选区
3. 操作按钮区
4. 列表表格区

### 8.3.3 功能要求

1. 支持上传需求文件。
2. 支持文件格式：
   - `txt`
   - `md`
   - `docx`
   - `pdf`
3. 上传后支持调用 LLM 自动生成测试点。
4. 支持查看测试点生成状态：
   - 待解析
   - 已生成
   - 已确认
5. 支持测试点人工确认与编辑。
6. 支持需求版本管理。
7. 支持按条件筛选需求记录。
8. 支持导出需求清单。

### 8.3.4 筛选项建议

1. 需求名称
2. 需求来源
3. 解析状态
4. 创建日期

### 8.3.5 操作按钮建议

1. 上传需求
2. LLM 生成测试点
3. 查看详情
4. 导出清单

### 8.3.6 列表字段建议

1. 序号
2. 状态
3. 需求名称
4. 上传文件
5. 版本
6. 生成测试点数
7. 更新时间
8. 负责人

### 8.3.7 LLM 输出要求

LLM 生成测试点时，至少应输出以下内容：

1. 测试点名称
2. 所属功能
3. 场景类型
4. 风险级别
5. 业务规则说明
6. 是否建议生成自动化用例

## 8.4 用例设计模块

### 8.4.1 功能目标

基于已确认的测试点，调用 LLM 自动生成测试用例，并支持人工审核与流转。

### 8.4.2 页面形式

采用列表页结构，包含：

1. 条件筛选区
2. 操作按钮区
3. 用例列表区

### 8.4.3 功能要求

1. 支持选择一个或多个测试点。
2. 支持基于选定测试点调用 LLM 生成测试用例。
3. 支持按正常、异常、边界场景生成测试用例。
4. 支持人工编辑、审核、删除测试用例。
5. 支持标记是否可自动化。
6. 支持将测试用例流转到接口自动化模块。
7. 支持按条件筛选测试用例。
8. 支持导出测试用例列表。

### 8.4.4 筛选项建议

1. 测试点
2. 用例状态
3. 是否可自动化
4. 创建日期

### 8.4.5 操作按钮建议

1. 选择测试点
2. LLM 生成用例
3. 编辑用例
4. 流转接口自动化

### 8.4.6 列表字段建议

1. 序号
2. 状态
3. 测试点
4. 用例标题
5. 优先级
6. 是否可自动化
7. 创建人
8. 最后更新时间

### 8.4.7 测试用例字段要求

每条用例至少包含：

1. 用例编号
2. 用例标题
3. 关联测试点
4. 前置条件
5. 测试步骤
6. 预期结果
7. 优先级
8. 是否可自动化
9. 审核状态

## 8.5 接口自动化模块

### 8.5.1 功能目标

基于指定测试用例执行接口自动化，返回执行结果，并提供 Allure 报告入口。

### 8.5.2 页面形式

采用列表页结构，包含：

1. 条件筛选区
2. 操作按钮区
3. 接口执行列表区

### 8.5.3 功能要求

1. 支持选择一个或多个测试用例。
2. 支持配置执行环境。
3. 支持后台自动组装并执行 HTTP 请求。
4. 支持规则断言。
5. 支持调用 LLM 对响应内容进行辅助分析。
6. 支持返回执行状态、结果详情和错误信息。
7. 支持失败重跑。
8. 支持查看 Allure 报告。

### 8.5.4 执行逻辑要求

建议执行链路如下：

1. 读取测试用例
2. 获取环境配置
3. 组装请求参数
4. 发送接口请求
5. 执行规则断言
6. 调用 LLM 进行结果分析
7. 保存执行结果
8. 生成 Allure 报告

### 8.5.5 筛选项建议

1. 测试用例
2. 执行环境
3. 执行状态
4. 执行日期

### 8.5.6 操作按钮建议

1. 选择测试用例
2. 批量执行
3. 查看结果
4. 打开 Allure
5. 失败重跑

### 8.5.7 列表字段建议

1. 序号
2. 状态
3. 测试用例
4. 执行环境
5. 接口名称
6. LLM 分析
7. 执行结果
8. 报告入口

### 8.5.8 执行结果字段

每条执行记录至少包含：

1. 执行任务编号
2. 测试用例编号
3. 执行环境
4. 请求地址
5. 请求参数
6. 响应结果
7. 断言结果
8. LLM 分析结果
9. 执行状态
10. 执行时间
11. 报告地址

## 9. Allure 报告需求

### 9.1 功能目标

统一展示接口自动化执行报告。

### 9.2 功能要求

1. 每次执行任务后生成 Allure 报告。
2. 支持在平台内通过链接或按钮打开报告。
3. 报告中至少展示：
   - 用例名称
   - 执行状态
   - 请求信息
   - 响应信息
   - 断言结果
   - 异常信息
   - 执行时间

### 9.3 附件建议

1. Request Headers
2. Request Body
3. Response Body
4. LLM 分析结果
5. 错误堆栈

## 10. LLM 能力要求

### 10.1 在需求设计模块

1. 读取需求文件内容
2. 提取测试点
3. 提取业务规则
4. 标记风险点

### 10.2 在用例设计模块

1. 基于测试点生成测试用例
2. 覆盖正常、异常、边界场景
3. 输出标准化用例结构

### 10.3 在接口自动化模块

1. 对执行结果进行语义分析
2. 对异常信息进行初步解释
3. 辅助判断是否存在业务风险

### 10.4 设计原则

1. 最终执行通过与失败，优先由规则断言决定。
2. LLM 主要承担生成、分析、解释与辅助判断能力。
3. LLM 输出应支持保存，便于审计与复盘。

## 11. 页面与交互要求

### 11.1 风格要求

1. 整体风格采用浅色科技蓝。
2. 视觉方向偏“蓝天感”，明亮、清爽、不压抑。
3. 项目内页面以列表台账风格为主。
4. 避免模块说明卡过多，突出核心数据和操作区。

### 11.2 页面要求

1. 项目内页面左侧只保留模块导航。
2. 项目内页面左上角提供返回项目列表箭头。
3. 模块页面采用统一列表布局，保证一致性。
4. 列表页优先展示筛选、操作、表格，不堆积无关说明区域。

## 12. 数据对象关系

V1 核心数据关系如下：

1. 一个项目包含多个需求
2. 一个需求包含多个测试点
3. 一个测试点包含多个测试用例
4. 一个测试用例可以触发多次接口执行
5. 一次接口执行对应一条结果记录与一份报告

可抽象的核心实体：

1. 项目
2. 需求
3. 测试点
4. 测试用例
5. 执行任务
6. 执行结果
7. Allure 报告
8. LLM 调用记录

## 13. 非功能需求

### 13.1 技术要求

1. 前端采用 `Vue`
2. 后端采用 `Spring Boot`
3. 前后端分离架构
4. 接口风格采用 RESTful

### 13.2 性能要求

1. 页面普通查询操作响应时间建议在 3 秒内。
2. LLM 生成任务与批量执行任务采用异步方式处理。
3. 列表页支持分页查询。

### 13.3 安全要求

1. 登录态校验
2. 项目级数据隔离
3. 操作日志记录
4. LLM 输入输出可追溯

### 13.4 可维护性要求

1. 模块化设计
2. 支持后续扩展 Web 自动化
3. 支持后续扩展更多模型或提示词模板

## 14. 验收标准

V1 完成的基本验收标准如下：

1. 用户可以通过登录页进入项目列表页。
2. 用户可以点击项目进入项目工作台。
3. 项目内可以看到需求设计、用例设计、接口自动化三个模块。
4. 需求设计模块支持上传需求并生成测试点。
5. 用例设计模块支持选择测试点并生成测试用例。
6. 接口自动化模块支持选择测试用例并执行接口任务。
7. 执行后可以查看执行结果和 Allure 报告入口。
8. 三个模块页面均采用统一列表台账式页面结构。

## 15. 二期规划

V2 规划如下：

1. Web 自动化
2. 更细粒度权限控制
3. 复杂流程编排
4. 多环境与多版本对比
5. 更完整的执行统计与分析能力
6. 更多自动化执行引擎能力

## 16. 附录

### 16.1 V1 一句话定义

智能测试平台 V1 是一个基于 `Vue + Spring Boot + LLM` 的项目化测试工作台，围绕“需求设计、用例设计、接口自动化”三大模块，打通从需求文件上传到接口自动化执行与 Allure 报告输出的核心闭环。
